# PARA Methodology / PARA 方法論

> [!info] 方法論概覽
> PARA 是一個由 [Tiago Forte](https://fortelabs.co/) 開發的生產力方法論，用於組織個人知識和任務。PARA 是 **Projects**、**Areas**、**Resources**、**Archives** 的縮寫。

---

## 📖 方法論介紹 / Introduction

### PARA 是什麼？

PARA 是一個基於**行動性**而非主題性的組織方法。它根據資訊的**當前相關性**和**使用頻率**來組織，而非根據內容的抽象分類。

> [!quote] 核心理念
> 「不要按主題組織，要按行動組織。」—— Tiago Forte

### 為何使用 PARA？

| 傳統方法 | PARA 方法 |
|---------|-----------|
| ❌ 按主題分類（如「商業」資料夾） | ✅ 按行動性質分類（專案、領域、資源、歸檔） |
| ❌ 混合活躍與不活躍的檔案 | ✅ 將活躍與不活躍項目分開 |
| ❌ 難以找到當前需要的資訊 | ✅ 快速定位當前相關的資訊 |
| ❌ 系統隨時間變得混亂 | ✅ 系統隨時間保持清晰 |

## 🎯 四大類別 / The Four Categories

### 1. Projects（專案）📌

**定義**：短期的、有明確目標和截止日期的努力。

**特徵**：
- ⏰ **有截止日期**：必須在某個時間點完成
- 🎯 **有明確結果**：可以定義「完成」的標準
- 👥 **有人負責**：通常是你，但可能涉及他人
- 📦 **有範圍**：有清晰的邊界

**範例**：
- 「推出新網站」
- 「完成年度稅務申報」
- 「學習 Python 基礎（3 個月）」
- 「搬遷到新住處」
- 「寫完產品白皮書」

**文件命名格式**：
```
2026-01-19 推出新網站.md
2026-Q1 學習 Python 基礎.md
```

**何時歸檔**：
- ✅ 專案完成時
- ✅ 專案取消/擱置時
- ✅ 專案轉變為長期責任（移至 Areas）

---

### 2. Areas（領域）🎯

**定義**：沒有截止日期的長期責任或持續性責任。

**特徵**：
- ♾️ **無截止日期**：持續進行，永遠不會「完成」
- 🔄 **需要定期維護**：需要持續關注
- 🎖️ **代表標準**：維持一定的生活或工作標準
- 📊 **可衡量進展**：可以追蹤當前狀態

**範例**：
- 「健康」 - 維持身體健康
- 「財務」 - 管理個人財務
- 「專業發展」 - 持續學習成長
- 「家庭關係」 - 維護家庭聯繫
- 「數字組織」 - 維護知識管理系統

**文件命名格式**：
```
Digital Organization.md
Health & Fitness.md
Personal Finance.md
```

**如何使用**：
- 設定維護標準和檢查清單
- 定期回顧（建議每月）
- 追蹤相關專案的進展

---

### 3. Resources（資源）📚

**定義**：你持續感興趣或正在研究的話題或主題。

**特徵**：
- 🔍 **主題導向**：按興趣主題組織
- 💡 **參考性質**：用於學習和參考
- 📈 **可能增長**：持續添加新資訊
- 🔗 **跨領域連結**：可連結到多個 Areas 和 Projects

**範例**：
- 「生產力技巧」
- 「料理食譜」
- 「行銷研究」
- 「AI 工具與應用」
- 「投資策略」

**文件命名格式**：
```
AI Tools & Applications.md
Productivity Tips.md
Cooking Recipes.md
```

**整理原則**：
- 使用標籤進行細分類
- 建立「主題索引」便於瀏覽
- 定期清理過時資訊

---

### 4. Archives（歸檔）📦

**定義**：已完成或不活躍的項目。

**特徵**：
- ⚰️ **不活躍**：當前不使用
- 📋 **已完成**：專案已結束
- 🔒 **保存價值**：保留以備未來參考
- 🗄️ **定期清理**：可刪除無用檔案

**範例**：
- 2024 年已完成的專案
- 舊的學習筆記（已被新知識取代）
- 過期的活動資訊
- 取消的專案資料

**歸檔策略**：
- 按年份或類別組織歸檔
- 設定定期清理時間（建議每季度）
- 考慮壓縮或刪除超過 1 年未使用的檔案

## 🔄 遷移流程 / Migration Flow

### 如何決定資訊歸類？

```
收件箱 (InBox)
    ↓
是否當前行動？
    ├── 否 → 歸檔 (Archives)
    └── 是 → 是否短期專案？
              ├── 是 → 專案 (Projects)
              └── 否 → 是否長期責任？
                        ├── 是 → 領域 (Areas)
                        └── 否 → 資源 (Resources)
```

### 決策流程圖

> [!flowchart] 分類決策樹
> ```
> 開始
>   │
>   ├─ 是否當前活躍？
>   │   ├─ 否 → 歸檔
>   │   └─ 是 → 是否有截止日期？
>   │       ├─ 是 → 專案
>   │       └─ 否 → 是否持續性責任？
>   │           ├─ 是 → 領域
>   │           └─ 否 → 資源
> ```

### 範例判斷

| 項目 | 當前活躍 | 截止日期 | 持續性 | 分類 |
|------|---------|---------|--------|------|
| 推出新網站 | ✅ | ✅ | ❌ | Projects |
| 健康管理 | ✅ | ❌ | ✅ | Areas |
| 投資研究 | ✅ | ❌ | ❌ | Resources |
| 2024 年度計畫 | ❌ | ❌ | ❌ | Archives |

## 📋 回顧週期 / Review Cycles

### PARA 回顧頻率建議

| 類別 | 回顧頻率 | 回顧重點 |
|------|---------|---------|
| **Projects** | 週回顧 | 進展、下一步行動、是否完成 |
| **Areas** | 月回顧 | 維護狀況、是否需要新專案 |
| **Resources** | 季回顧 | 相關性、是否需要清理 |
| **Archives** | 年回顧 | 清理、壓縮、刪除無用檔案 |

- [ ] 專案 B - 遇到障礙
- [ ] 專案 C - 已完成 → 移至歸檔

### 本週完成
-

-

-
```

## 🎓 最佳實踐 / Best Practices

### ✅ DO（應該做的事）

1. **單一原則**
   - 每個檔案只放在一個類別中
   - 使用標籤和連結進行關聯

2. **快速決策**
   - 新資訊在 2 分鐘內分類
   - 不要過度思考

3. **定期清理**
   - 每週整理收件箱
   - 每月檢視 Areas 和 Resources
   - 每季度整理歸檔

4. **命名一致**
   - 使用一致的檔案命名格式
   - 包含日期、專案名稱等關鍵資訊

### ❌ DON'T（不該做的事）

1. **避免過度細分**
   - 不要創建過多的子類別
   - 保持結構簡單

2. **不要混合活躍度**
   - 活躍項目與非活躍項目分開
   - 專案不要與長期資源混在一起

3. **不要等待「完美系統」**
   - 從簡單開始
   - 隨著使用逐步優化

4. **不要忽略歸檔**
   - 完成的專案立即歸檔
   - 定期清理無用檔案

## 📌 常見問題 / FAQ

### Q1: 某個項目既像專案又像領域，如何分類？

**A**: 判斷標準是「是否有明確的完成時間」。
- 如果有截止日期 → 專案
- 如果持續進行，永遠不會「完成」→ 領域

### Q2: 學習一個技能是專案還是資源？

**A**: 取決於學習的方式。
- 有明確學習目標和時間（如「3 個月學完 Python」）→ 專案
- 持續收集學習資料和筆記 → 資源
- 維持技能作為長期責任 → 領域

### Q3: 如何處理跨多個類別的筆記？

**A**: 使用連結和標籤。
- 筆記放在主要類別（最相關的）
- 使用 wikilinks 連結到其他相關類別
- 使用標籤標記多個分類

### Q4: PARA 適合團隊使用嗎？

**A**: 可以，但需要適配。
- 團隊使用共享 Projects 和 Areas
- 個人維護自己的 Resources 和 Archives
- 建立團隊約定的命名和分類規範
